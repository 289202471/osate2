Error-Model Annex v2 - Unhandled Faults

Rationale

When reusing components in an architecture, we may reuse their related error description. However, when reusing the component, some faults propagated may not be handled by the components connected to it. Also, components that are connected may expect to receive fault types that are not propagated.

Using the Unhandled Fault function

To use the function, select your instance model and select the UnhandledFaults function in the Fault Analysis menu, as shown below.

Type of faults under analysis

  • Components with error source not connected to any error path
  • Components with error sink not connected to any error source
  • Coverage of all fault types across error propagations. For example, if the model defines fault types TimingError with two subtypes: LateDelivery and EarlyDelivery. Then, when having a component propagating LateDelivery, the tool must make sure that the connected component handled this type or the super-type TimingError. Also, the tool must check that the receiving component handle all error sub-types: EarlyDelivery and LateDelivery. In other words, it checks that all types of faults are handled.
